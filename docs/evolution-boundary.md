# Evidence-Gated Asset Evolution boundary

`aet learn` turns reproducible, structured failures into bounded asset
candidates. It is not arbitrary source-code self-modification. The production
asset changes only after a target-specific Gate, optional Shadow evidence, and
an explicit human `adopt --yes` decision.

## Evolution Constitution

Every Candidate IR v2 binds the target type and path, baseline and candidate
hashes, one to three allowlisted operations, adoption mode, source patterns,
and the canonical Constitution hash. The Constitution is outside all candidate
write surfaces and enforces:

- `UNKNOWN` is not a pass and natural language is not execution proof;
- candidates cannot modify evaluators, scoring, held-out data, status semantics,
  the Constitution, or the human-adoption requirement;
- validation and held-out cases are distinct;
- a passing Gate is not adoption, commit, push, upload, or scheduling;
- the production target must still match the recorded baseline hash;
- Shadow Audit cannot affect official findings or exit status.

Legacy Skill Candidate v1 manifests are upgraded only in memory. New proposals
use `evolution-candidate/v2`; a non-Skill v1 target is never inferred.

## Supported targets

| Target | Allowed change | Evaluator |
| --- | --- | --- |
| `skill` | Existing named editable blocks | Static contract; opt-in real-host paired rollout for observed Agent behavior. |
| `audit-rule` | Non-executable RulePack entries using allowlisted detectors | Core/validation/held-out/adversarial audit fixtures and candidate-bound Shadow Audit. |
| `audit-profile` | Severity and sensitive-path tightening; only pre-approved exclusions may remain | Deterministic policy suites and runtime `audit --profile`. |
| `review-policy` | Smaller path budget, added sensitive paths, added proof requirements | Deterministic policy suites and runtime `review --policy`. |
| `trace-validator` | Tighter JUnit/SARIF/coverage/JSON assertions | Safe parser over a fresh, explicitly declared Trace artifact. |
| `triage-policy` | Non-negative weights and critical paths | Deterministic ordering; finding status and visibility are immutable. |

The RulePack is a versioned safe detector DSL, not an arbitrary code plugin.
Built-in findings declare their own ID, status, severity, claim, remediation,
target kinds, revision, and allowlisted detector. The detector implementation
remains trusted evaluator code and cannot be generated by a candidate.

## Evidence and feedback

Evidence Only is the default. Harvest and local federation retain report,
finding, artifact, snapshot, task, runner, and decision hashes—not raw prompts,
transcripts, shell output, secrets, or environment variables.

Audit feedback is adoption-grade only when a local fixture reproduces the
declared false positive, false negative, wrong status/severity/location,
incomplete remediation, duplicate finding, non-determinism, performance
regression, or explicit policy-exception case. The expected manifest and full
fixture tree are hashed. Subjective feedback without a reproducer fails closed.

## Target-specific Gates

Skill static replay is Gate 0 and uses `static_contract_*` metrics. Scripted,
Codex, and Claude Code runners are opt-in and apply only to Skill behavior.
They use isolated fixture copies, normalized events, actual command/artifact
evidence, deterministic scoring, paired statistics, and explicit
`INCONCLUSIVE` / `INFRASTRUCTURE_ERROR` states.

Audit Rule Gate uses 30 core, 15 validation, 15 held-out, and 10 adversarial
tasks across 22 reusable fixture repositories. Validation and held-out cannot
reuse a fixture or identical case. A candidate must improve validation without
core/held-out regression, new unapproved findings, increased false positives
or protected false negatives, determinism loss, or runtime-budget breach.

Audit Rule adoption also requires an aggregate bound to the same candidate
hash with at least 20 Shadow runs, 5 repository fingerprints, and 3 dates.
Every added finding must be confirmed and confirmed false positives must be
zero. Tests exercise this contract with synthetic aggregates; the repository
does not claim that a release candidate has already accumulated real multi-day
Shadow evidence.

Each policy target uses separate core, validation, held-out, and adversarial
suites. Gate results are vectors, not one agent-trust score. Only `PASS` can
stage; `FAIL`, `INCONCLUSIVE`, and `INFRASTRUCTURE_ERROR` cannot.

## Stage, adopt, and Sleep

Stage copies a hash-verified candidate for review. Adopt rechecks candidate,
Gate, Constitution and current baseline, prepares a Decision Ledger update,
atomically replaces the target, and rolls back if the ledger cannot be
finalized. It never commits or pushes.

Sleep remains stage-only. For non-Skill targets it uses the registered adapter
and target-specific evaluator; it never silently falls back to a Skill Gate.
No AET command installs a scheduler, uploads experiences, logs in to a host, or
automatically adopts a candidate.
